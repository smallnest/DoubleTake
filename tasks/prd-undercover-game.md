# PRD: 谁是卧底 — 命令行网络游戏

## 1. Introduction/Overview

用 Go 语言实现一个基于 TCP socket 的命令行多人网络游戏"谁是卧底"。程序设计为摸鱼场景使用，外观伪装成普通命令行工具（如系统诊断、日志分析等），不使用任何花哨的 TUI 库。一个二进制文件通过命令行参数区分裁判（服务端）和玩家（客户端）角色，支持 4-10 人同时在线游戏。

核心游戏流程：裁判创建房间 → 生成密码（编码了 IP:port） → 玩家输入密码自动连接 → 分配词语和角色 → 轮流描述 → 投票淘汰 → 判断胜负。

## 2. Goals

- 实现完整的"谁是卧底"游戏逻辑（平民、卧底、白板三种角色）
- 基于 TCP socket 的网络通信，裁判端为服务端，玩家端为客户端
- 明文传输，简洁的文本协议
- 单一可执行文件，通过 `--role` 参数区分裁判/玩家模式
- 房间密码直接编码 IP:port，玩家输入密码即可自动连接
- 支持断线重连（玩家可通过名字+密码恢复身份）
- 命令行界面伪装成系统工具风格，不引人注目

## 3. User Stories

### US-001: 裁判启动服务端并创建房间
**Description:** 作为裁判，我想通过命令行启动服务端并配置游戏参数，以便创建一个游戏房间等待玩家加入。
**Acceptance Criteria:**
- [ ] 执行 `doubletake --role judge` 启动裁判模式，交互式引导输入玩家人数(4-10)、卧底人数(1-3)、白板人数(0-N)
- [ ] 校验参数合法性：卧底+白板人数 < 平民人数，总人数 4-10
- [ ] 自动获取本机 IP，结合端口生成房间密码（将 `IP:port` 编码为 base62 格式的短字符串）
- [ ] 在控制台显示房间密码，提示等待玩家加入
- [ ] TCP 服务端监听指定端口，准备接受玩家连接
- [ ] 编译通过，无 lint 错误

### US-002: 玩家通过密码自动连接并加入房间
**Description:** 作为玩家，我想输入一个密码就能自动连接到裁判并加入游戏，不需要手动输入 IP 和端口。
**Acceptance Criteria:**
- [ ] 房间密码由裁判的 `IP:port` 编码生成（base62 编码），玩家无需知道 IP 和端口
- [ ] 执行 `doubletake --role player` 后只需输入房间密码
- [ ] 客户端自动解码密码获取 IP:port 并建立 TCP 连接
- [ ] 连接成功后交互式引导输入玩家名字
- [ ] 服务端拒绝重名玩家，提示更换名字
- [ ] 裁判端实时显示已加入的玩家列表和人数
- [ ] 编译通过，无 lint 错误

### US-003: 裁判分配角色并分发词语
**Description:** 作为裁判，我想在所有玩家加入后输入两个词语并分配角色，以便游戏正式开始。
**Acceptance Criteria:**
- [ ] 当加入玩家数达到设定人数时，裁判端提示输入平民词语和卧底词语
- [ ] 裁判端也可以提前手动输入 `start` 触发（当人数 >= 4 时）
- [ ] 系统随机分配角色：平民拿到平民词语，卧底拿到卧底词语，白板收到"你是白板"提示
- [ ] 每个玩家客户端显示自己的词语（或白板提示），不显示角色名称
- [ ] 编译通过，无 lint 错误

### US-004: 玩家轮流描述词语
**Description:** 作为玩家，我想按顺序用一句话描述自己的词语，以便其他玩家判断身份。
**Acceptance Criteria:**
- [ ] 裁判宣布轮次开始，按加入顺序（或随机顺序）指定当前描述的玩家
- [ ] 非当前玩家看到提示"等待 XXX 描述..."
- [ ] 当前玩家输入描述文字后，所有玩家（和裁判）都能看到该描述
- [ ] 描述文字不能为空，服务端拒绝空描述
- [ ] 裁判端记录每轮所有玩家的描述
- [ ] 编译通过，无 lint 错误

### US-005: 投票环节
**Description:** 作为玩家，我想在描述环节结束后投票选出可疑玩家，以便淘汰卧底。
**Acceptance Criteria:**
- [ ] 所有玩家描述完毕后，裁判宣布进入投票环节
- [ ] 每个玩家看到存活玩家列表，输入要投票的玩家名字
- [ ] 玩家不能投自己，不能投已出局的玩家，服务端校验并提示重新输入
- [ ] 所有玩家投票完成后，裁判端统计并公布票数和结果
- [ ] 票数最高的玩家出局；平票时进入 pk 模式（平票玩家各再描述一次后单独投票），pk 允许多轮直到分出结果
- [ ] 编译通过，无 lint 错误

### US-006: 游戏胜负判定
**Description:** 作为系统，我需要在每轮结束后判定游戏是否结束及胜负，以便宣布结果。
**Acceptance Criteria:**
- [ ] 每轮淘汰后检查：若卧底全部出局 → 平民胜利
- [ ] 每轮淘汰后检查：若卧底人数 >= 平民人数 → 卧底胜利
- [ ] 白板玩家在任何时候可以输入 `guess <词语>` 猜测平民词语，猜对则白板单独获胜
- [ ] 游戏结束时，裁判公布所有玩家的角色和双方词语
- [ ] 游戏结束后裁判端可以选择"再来一局"或退出
- [ ] 编译通过，无 lint 错误

### US-007: 断线重连机制
**Description:** 作为玩家，我想在意外断线后能重新连接并恢复游戏身份，以便不因网络问题被踢出。
**Acceptance Criteria:**
- [ ] 玩家断线后，裁判端标记该玩家为"掉线"状态，不自动踢出
- [ ] 玩家重新启动客户端，输入相同的密码和名字
- [ ] 服务端验证名字匹配后恢复该玩家的角色和词语
- [ ] 重连玩家获得当前游戏状态摘要（当前轮次、自己的词语）
- [ ] 如果轮到掉线玩家操作，裁判端等待直到超时（60秒）后自动跳过
- [ ] 编译通过，无 lint 错误

### US-008: 命令行伪装界面
**Description:** 作为摸鱼玩家，我希望程序的输入输出看起来像系统诊断工具，以便不被旁人发现。
**Acceptance Criteria:**
- [ ] 程序启动时不显示游戏标题或 banner，显示类似 `[sysmon] v2.4.1 — diagnostics mode` 的伪装提示符
- [ ] 输入提示符使用技术风格的格式，如 `[node-03]$ ` 而非 `请输入: `
- [ ] 游戏消息用类似日志输出的格式显示，如 `[INFO] [session:7a3f] data received from node-05`
- [ ] `--stealth` 参数可以进一步隐藏特征（可选功能，非必须）
- [ ] 编译通过，无 lint 错误

### US-009: TCP 通信协议
**Description:** 作为开发者，我需要定义简洁的文本协议用于裁判和玩家之间的通信。
**Acceptance Criteria:**
- [ ] 定义消息格式：`<TYPE>|<payload>\n`，如 `JOIN|alice\n`、`DESC|这是一个水果\n`
- [ ] 消息类型至少包括：JOIN, READY, ROLE, TURN, DESC, VOTE, RESULT, KICK, WIN, ERROR, RECONNECT
- [ ] 服务端广播类消息（描述、投票结果）发送给所有连接的玩家
- [ ] 私密消息（角色分配）仅发送给对应玩家
- [ ] 客户端异常断开时服务端不 panic，优雅处理连接关闭
- [ ] 编译通过，无 lint 错误

### US-010: 再来一局
**Description:** 作为裁判/玩家，我想在游戏结束后保留相同玩家直接开始新一局，除非有人主动退出。
**Acceptance Criteria:**
- [ ] 游戏结束后，裁判端提示"再来一局(Y/N)？"，输入 Y 则保留所有已连接玩家开始新一局
- [ ] 再来一局时，裁判重新输入两个词语，重新随机分配角色
- [ ] 玩家端自动显示新一轮的词语（或白板提示），无需重新连接
- [ ] 玩家可随时输入 `quit` 退出游戏，退出后裁判端标记该玩家已离开
- [ ] 裁判退出则整个房间关闭，所有玩家断开连接
- [ ] 编译通过，无 lint 错误

## 4. Functional Requirements

- FR-1: 系统必须支持通过 `--role judge` 启动裁判模式，`--role player` 启动玩家模式
- FR-2: 裁判模式必须支持交互式配置玩家人数(4-10)、卧底人数(1-3)、白板人数(0+)
- FR-3: 系统必须基于 TCP socket 实现 C/S 架构的通信
- FR-4: 系统必须将 `IP:port` 编码为 base62 短字符串作为房间密码，玩家输入密码即可解码并自动连接
- FR-5: 服务端必须验证玩家加入时的名字唯一性
- FR-6: 系统必须在游戏开始时随机分配角色（平民/卧底/白板）并私密发送对应词语
- FR-7: 系统必须按顺序让每个存活玩家描述词语，广播描述内容给所有玩家
- FR-8: 系统必须在描述结束后进入投票环节，每个玩家投票选择一名可疑玩家
- FR-9: 服务端必须统计投票结果，票数最高者出局；平票时进入 pk 再投票
- FR-10: 系统必须在每轮淘汰后判定：卧底全部出局则平民胜，卧底数>=平民数则卧底胜
- FR-11: 系统必须允许白板玩家通过 `guess <词语>` 猜测平民词语，猜对则白板胜
- FR-12: 系统必须支持断线重连，通过名字+密码验证身份并恢复游戏状态
- FR-13: 系统必须在掉线玩家操作时设置 60 秒超时自动跳过
- FR-14: 所有输入输出必须使用伪装成系统工具的格式
- FR-15: 通信协议必须使用 `<TYPE>|<payload>\n` 的文本格式
- FR-16: 游戏结束后必须支持"再来一局"，保留相同玩家连接，重新分配角色
- FR-17: 裁判退出必须关闭整个房间，所有玩家断开连接

## 5. Non-Goals (Out of Scope)

- 不做图形界面（TUI/GUI），纯命令行文本输入输出
- 不做 TLS 加密，明文传输
- 不做跨 NAT 打洞，裁判端需要公网 IP 或同局域网
- 不做游戏录像/回放功能
- 不做 AI 托管（玩家掉线不自动操作，仅跳过回合）
- 不做词语库内置，裁判自行输入词语
- 不做多房间支持，一个裁判实例只主持一局游戏

## 6. Design Considerations

- **界面伪装风格：** 输入提示符用 `[node-XX]$ `，消息用 `[INFO]`/`[WARN]`/`[DATA]` 前缀，列表用类似 `grep` 输出格式
- **端口选择：** 默认端口建议 `8127`（不常见端口），可通过 `--port` 参数修改
- **交互模式：** 全部使用标准 `fmt.Scan`/`bufio.Scanner` 读取输入，不依赖 cobra 等框架
- **密码编码：** 将 `IP:port` 字符串进行 base62 编码，生成短小、易分享的房间密码

## 7. Technical Considerations

- Go 标准库 `net` 包实现 TCP 通信
- Go 标准库 `encoding/base64` + 自定义 base62 实现密码编解码
- 每个客户端连接用独立 goroutine 处理
- 使用 `sync.Mutex` 保护共享状态（玩家列表、投票结果等）
- 使用 channel 协调游戏阶段转换（等待加入 → 描述 → 投票 → 判定）
- 消息以 `\n` 作为分隔符，`bufio.Scanner` 逐行读取
- 项目结构建议：
  ```
  doubletake/
  ├── main.go              # 入口，参数解析
  ├── cmd/
  │   ├── judge.go         # 裁判模式逻辑
  │   └── player.go        # 玩家模式逻辑
  ├── server/
  │   ├── server.go        # TCP 服务端
  │   ├── room.go          # 房间/游戏状态管理
  │   └── protocol.go      # 消息协议定义与解析
  ├── client/
  │   ├── client.go        # TCP 客户端
  │   └── display.go       # 伪装显示逻辑
  └── game/
      ├── game.go          # 游戏核心逻辑
      ├── role.go          # 角色定义与分配
      └── vote.go          # 投票逻辑
  ```

## 8. Success Metrics

- 4-10 人可以稳定完成一局完整游戏（从加入到分出胜负）
- 断线后 60 秒内可重连并继续游戏
- 单条消息延迟 < 500ms（局域网环境）
- 编译后单二进制文件 < 10MB
